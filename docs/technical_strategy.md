# 技术思路说明

## 核心问题

这个项目的核心问题不是“让模型随便写点创意”，而是：

1. 角色原始资产必须稳定，不能漂移。
2. 创意内容必须持续发散，不能坍缩到少数高频套路。
3. 下游输出必须能稳定转成可执行的生图请求。

## 坍缩是怎么发生的

LLM 本质上是一个概率预测机。人物原始资产越稳定，输入分布越容易收缩到少数熟路数。
如果没有外部变量，创意层会不断回落到同一个概率盆地，最终表现为：

- 场景越来越像
- 动作越来越像
- 情绪组织方式越来越像
- 构图和服装越来越模板化

## 坍缩的解决方法

当前方案不是“让模型更随机”，而是同时做两件事：

1. 给创意层引入外部发散变量  
   通过采样层从外部内容里抽取同源 shortlist，再由过滤器从 shortlist 中选出一条信号。

2. 让创意产出对外部变量保持锚定  
   采样信号不是装饰性参考，而是正式输入。场景设计稿必须吸收这条外部变量，形成有来源的世界状态。

只有“输入发散”而没有“输出锚定”，结果会变成随机漂移。  
只有“输出锚定”而没有“输入发散”，结果会继续坍缩。

## 当前链路的技术判断

当前链路的职责划分是：

- `creative` 负责把“稳定角色 + 外部变量”转成三份可消费设计
- `social_post` 负责把“稳定角色 + 场景设计稿”转成主角口吻的社媒文案
- `prompt_builder` 负责把三份设计编译成生图模型可消费的 prompt
- `execution` 负责把 prompt 注入 ComfyUI workflow 并稳定出图

这三层里，真正负责“防坍缩”的是 `creative`。  
`prompt_builder` 和 `execution` 不是用来制造新创意的，它们负责编译和执行。

## 当前已知风险

当前系统已经能稳定跑通整条链，但内容质量上还有一个明确风险：

- `prompt_builder` 容易把多份设计里的细节并列堆进 prompt
- 当不同设计对景别、姿态、可见身体部位提出互相顶牛的要求时，模型会生成失真的肢体或弱化关键锚点

这不是执行层不稳定，而是 prompt 编译层还不够会做“约束裁决”。

## 当前执行策略

所以现阶段的技术策略是：

1. 保持 `creative -> prompt_builder -> execution` 的单向链路
2. 不让 execution 层重新发明内容
3. 在 creative 和 prompt_builder 之间逐步收紧“冲突约束的裁决规则”
4. 用真实出图结果反推 prompt 编译质量，而不是把问题甩给工作流
