# 技术策划

`techart-agent` 的产品目标是为 galgame 开发者提供本地技术美术资产生产客户端。它以角色资产库为中心，通过本地 ComfyUI 工作流生产依赖资产、扩展子资产，并在资产约束下生成成品插画。

## 三个主功能模块

### 角色设计期资产生成与筛选

这个模块服务角色从概念走向技术美术起点。用户可能只有文字设定、项目画风、少量草图或风格参考。产品要帮助用户生成一批可筛选的基础资产，并把通过审核的结果写入角色资产库。

输入：

- 角色文字设定
- 项目美术标准
- 少量草图或参考图
- 服装方向
- 姿态方向
- 资产类型目标

输出：

- 面部标准图
- 三分之二脸
- 全身正面
- 全身侧面
- 全身背面
- 表情表候选
- 发型细节
- 眼睛细节
- 配饰细节
- 服装正面、侧面、背面
- 姿态参考图
- 已通过样例
- 失败样例

产品行为：

```text
用户选择资产目标
  -> 系统生成候选
  -> 用户筛选
  -> 入库到对应资产包
  -> 失败图记录问题类型
```

验收标准：

- 生成结果能否作为后续工作流的稳定依据。
- 图像是否能清楚表达角色身份、比例、发型、服装或细节。
- 通过图和失败图是否都有清楚归档位置。

### 依赖资产延展

这个模块服务角色资产从少量标准图扩展到完整生产资料。用户已经有基础设定图、面部标准、服装标准和部分通过样例，产品要继续生成更多子资产。

输入：

- 已有角色资产
- 指定资产类型
- 指定服装
- 指定表情、姿态、动作或局部细节
- 项目美术标准

输出：

- 表情差分
- 姿态参考
- 服装差分
- 配饰差分
- 细节特写
- 不同视角图
- 不同场景状态图
- 局部修复样例
- 训练数据候选

产品行为：

```text
用户选择已有角色与扩展目标
  -> 资产选择器读取 reference set
  -> 本地工作流生成子资产候选
  -> 用户筛选
  -> 写入对应资产包
```

验收标准：

- 子资产是否保持角色识别度。
- 子资产是否能继续作为后续生成依赖。
- 服装、配饰、表情和姿态是否能被明确调用。

### 成品插画生成

这个模块服务最终图片生产。用户基于已入库的角色资产、服装资产和姿态/构图需求，生成事件 CG、宣传插画、章节图和角色单人图。

输入：

- 角色资产库
- 服装资产
- 配饰资产
- 姿态或构图需求
- 事件描述
- 场景说明
- 项目美术标准
- 导出规格

输出：

- 候选插画
- 精修插画
- 局部修复版本
- 高清导出图
- 运行记录
- 审图记录

产品行为：

```text
用户创建插画任务
  -> 系统选择依赖资产
  -> 本地工作流生成候选图
  -> 用户选择候选图
  -> 精修或局部修复
  -> 验收
  -> 入库与导出
```

验收标准：

- 角色是否像。
- 服装是否对。
- 事件是否成立。
- 姿态和构图是否接近需求。
- 局部是否存在明显崩坏。
- 输出尺寸和格式是否符合项目交付要求。

## 共享角色资产库

三个模块共用 Character Tech-Art Kit。角色设计期向资产库写入基础资产；依赖资产延展继续扩充资产库；成品插画生成从资产库读取依赖，并把通过图、失败图和审图记录回流到 `quality/`。

```text
角色设计期资产生成
  -> 写入 model_sheets、detail_sheets、outfits、pose_sheets

依赖资产延展
  -> 写入 expressions、pose_sheets、detail_sheets、approved_examples

成品插画生成
  -> 读取 reference_sets
  -> 写入 candidates、refinements、inpaint、exports、quality
```

## 任务类型

第一批任务类型建议：

```text
asset_bootstrap.face
asset_bootstrap.turnaround
asset_bootstrap.outfit
asset_bootstrap.detail
asset_extend.expression
asset_extend.pose
asset_extend.outfit_variant
asset_extend.detail_variant
illustration.event_cg
illustration.standing
illustration.promo
```

每个任务类型需要绑定：

- 默认工作流 preset
- 默认 reference set
- 默认审图清单
- 默认导出规格
- 可选模型资源

## 产品与本地工作流对应

| 产品动作 | 本地工作流 | 调用资产 | 成熟度判断 |
| --- | --- | --- | --- |
| 生成角色面部标准图 | reference/image-edit candidate | 项目画风、少量草图、文字设定 | 需要实测 |
| 生成全身正面或三视图候选 | controlled candidate | 文字设定、草图、比例参考、服装方向 | 需要实测 |
| 生成服装设定候选 | reference/image-edit candidate | 角色全身图、服装草图、材质说明 | 需要实测 |
| 扩展表情差分 | image-to-image / reference edit | 面部标准图、表情要求、通过样例 | 可落地，需要比较模型 |
| 扩展姿态参考 | ControlNet pose / lineart | 姿态草图、角色参考、服装参考 | 可落地 |
| 生成事件 CG 候选 | candidate_controlled | 面部身份、三视图、服装、姿态、构图 | 可落地，需要质量验证 |
| 精修选中图 | img2img refine | 选中图、面部身份、服装细节 | 成熟 |
| 局部修复 | inpaint_region | 源图、遮罩、细节图、修复要求 | 成熟 |
| 高清导出 | export_upscale | 通过验收图、导出规格、upscale 配置 | 成熟 |

这里的“成熟”表示 ComfyUI 工作流能力本身清楚。实际效果仍然要用本项目角色资产进行验证。

## MVP

第一版 MVP 只做最短闭环：

```text
角色设计期：
  面部标准图、全身正面、服装正面

依赖资产延展：
  表情差分、姿态参考

成品插画：
  事件 CG 候选、精修、局部修复、高清导出
```

MVP 的目标是验证“少量角色资产能否驱动本地工作流稳定产出可筛选素材”。验证通过后，再扩展三视图、服装背面、复杂配饰、表情表、宣传图和批量导出。

## 实测矩阵

第一轮技术验证必须跑矩阵，不直接承诺效果。

```text
A. 基础 prompt + checkpoint
B. 参考图适配 + prompt
C. 参考图适配 + OpenPose
D. 参考图适配 + Lineart / Depth
E. 图像编辑模型候选
F. 选中图 img2img 精修
G. 局部 inpaint 修脸、手、服装细节
H. upscale 导出
```

同一角色、同一服装、同一事件下比较：

- 角色脸稳定性
- 发型稳定性
- 体型比例稳定性
- 服装结构保留
- 姿态遵循
- 构图遵循
- 局部修复成功率
- 单张耗时
- 失败率

这组验证决定第一版默认工作流。
